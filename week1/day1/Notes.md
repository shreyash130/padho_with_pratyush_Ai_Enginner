# **Understand Your First LLM Call**

# Virtual Environment (`venv`)

## Definition

A **Virtual Environment** is an isolated Python environment that allows each project to have its own Python packages and dependency versions without affecting other projects or the global Python installation.

## Why Do We Need a Virtual Environment?

Different projects may require different versions of the same library.

### Example

- **Project A** requires `numpy==1.24`
- **Project B** requires `numpy==2.0`

If we install or upgrade NumPy globally, one of the projects may stop working.

A virtual environment solves this problem by creating an **isolated environment for each project**.

## Important Point

> `venv` is mainly used to isolate **Python packages and dependencies**, not hardware drivers.

## Create a Virtual Environment

```bash
python -m venv venv
```

Here:

- `python` → Runs Python
- `-m venv` → Uses Python's built-in `venv` module
- `venv` → Name of the virtual environment folder
- 
## Activate a Virtual Environment

### Windows

```bash
venv\Scripts\activate
```

### Linux / macOS

```bash
source venv/bin/activate
```

## Deactivate a Virtual Environment

```bash
deactivate
```

# LLM API Call

## Definition

An **LLM API call** is a request sent from an application to a Large Language Model service to generate a response based on the provided instructions and input.

## Basic Flow

```text
Application / Client
        |
        |  API Request
        v
    LLM Server
        |
        |  Model Processing
        v
    API Response
        |
        v
Application / Client
```

## Main Components of an LLM API Call

### 1. API Key

An **API key** is a secret credential used to authenticate the application with the LLM service.

```text
Client/Application → API Key → LLM Server
```

### Important Points

- The API key identifies and authenticates the application.
- It should be kept secret.
- It should not be exposed in public repositories or frontend code.
- It is commonly stored in environment variables or a `.env` file.

Example:

```env
API_KEY=your_api_key_here
```

---

### 2. Model

The **model** specifies which LLM should process the request.

The exact model identifier must be provided according to the API being used.

Conceptual example:

```python
model="model-name"
```

Different models may differ in:

- Intelligence and reasoning ability
- Speed
- Cost
- Context window
- Multimodal capabilities

---

### 3. Messages

Messages contain the conversation or instructions sent to the LLM.

A message generally contains:

```text
Message
├── Role
└── Content
```

Example:

```python
{
    "role": "user",
    "content": "Explain machine learning."
}
```

---

# Roles in an LLM Conversation

## 1. System Role

The **system role** defines the overall behavior, rules, personality, or instructions for the AI.

Example:

```python
{
    "role": "system",
    "content": "You are a helpful machine learning tutor."
}
```

### Purpose

- Sets behavior
- Defines rules
- Provides high-level instructions
- Controls response style

---

## 2. User Role

The **user role** represents the input or question provided by the user.

Example:

```python
{
    "role": "user",
    "content": "What is supervised learning?"
}
```

---

## 3. Assistant Role

The **assistant role** represents responses previously generated by the AI.

Example:

```python
{
    "role": "assistant",
    "content": "Supervised learning is a type of machine learning..."
}
```

Previous assistant messages can be included to maintain conversation context.

---

# Content

The **content** contains the actual text, instruction, question, or other supported input provided in a message.

Example:

```python
{
    "role": "user",
    "content": "Explain linear regression."
}
```

Here:

- `role` = Who sent the message
- `content` = What the message contains

---

# LLM API Response

After processing the request, the LLM API returns a **response object**.

A response may contain information such as:

```text
Response
├── Generated Output / Answer
├── Usage
│   ├── Input Tokens
│   ├── Output Tokens
│   └── Total Tokens
└── Other Metadata
```

The exact response structure depends on the API and SDK being used.

---

# Usage

The `usage` section provides information about the number of **tokens** consumed by the request.

It may contain:

```text
Usage
├── Input Tokens
├── Output Tokens
└── Total Tokens
```

## Token

A **token** is a small unit of text processed by an LLM.

A token can be:

- A complete word
- Part of a word
- Punctuation
- A symbol

### Why Are Tokens Important?

Tokens affect:

- API cost
- Context limits
- Input size
- Output size

---

# Choices

In some LLM APIs, generated responses are returned inside a `choices` array.

Conceptual structure:

```text
Response
└── Choices
    ├── Choice 1
    ├── Choice 2
    └── ...
```

Each choice may contain a generated answer.

> Important: Not every modern LLM API uses the same response structure. Some APIs use `choices`, while others use different output structures.

---

# Complete Conceptual Flow

```text
1. Create / Configure Client
        |
        v
2. Provide API Key
        |
        v
3. Select Model
        |
        v
4. Send Messages
        |
        ├── System Role
        ├── User Role
        └── Assistant Role
        |
        v
5. LLM Processes Input
        |
        v
6. Receive Response
        |
        ├── Generated Answer
        ├── Token Usage
        └── Metadata
```

# LLM API Call

## `.env`

Store the API key in a `.env` file:

```env
GROQ_API_KEY=your_api_key_here
```

> Never expose or upload your API key to GitHub.

---

## `hello_llm.py`

```python
import os
from dotenv import load_dotenv
from groq import Groq

# Load variables from the .env file
load_dotenv()

# Get the API key
my_api_key = os.getenv("GROQ_API_KEY")

# Check if the API key exists
if not my_api_key:
    raise ValueError("GROQ_API_KEY is missing")

# Create the Groq client
client = Groq(api_key=my_api_key)

# Select the model
model = "llama-3.3-70b-versatile"

# Create the prompt
prompt = "What is Machine Learning?"

# Send the request to the LLM
response = client.chat.completions.create(
    model=model,
    messages=[
        {
            "role": "user",
            "content": prompt
        }
    ]
)

# Get the main answer
answer = response.choices[0].message.content

print(answer)
```

# Important Terms

**API Key:** Secret key used to authenticate with the API.

**Client:** Object used to communicate with the LLM API.

**Model:** The specific LLM used to generate the response.

**Role:** Defines who sent the message, such as `user`, `assistant`, or `system`.

**Content:** The actual text or prompt sent to the LLM.

**Response:** The complete result returned by the API.

**Choices:** Contains the generated response options.

# Quick Revision Questions

## Q1. What is an API key?

An API key is a secret credential used to authenticate an application when accessing an API.

## Q2. What is the purpose of specifying a model in an LLM API call?

The `model` parameter tells the API which Large Language Model should process the request.

## Q3. What are roles in an LLM conversation?

Roles identify the purpose or sender of each message in a conversation. Common roles are `system`, `user`, and `assistant`.

## Q4. What is the system role?

The `system` role provides high-level instructions that define how the AI should behave or what role it should follow.

## Q5. What is the user role?

The `user` role represents the user's input, question, or request sent to the LLM.

## Q6. What is the assistant role?

The `assistant` role represents messages generated by the AI. Previous assistant messages can also be included in the conversation history to maintain context.

## Q7. What is a token?

A token is a unit of text processed by a language model. Tokens are used to measure input size, output size, context usage, and often API consumption.


## Q8. What is the difference between input tokens and output tokens?

- **Input tokens** → Tokens sent to the model.
- **Output tokens** → Tokens generated by the model.
- **Total tokens** → Input tokens + Output tokens.

## Q9. What is an LLM API call?

An LLM API call is a request sent from an application to an LLM service containing input and configuration, which the model processes to generate a response.

## Q10. Why do we use environment variables for API keys?

Environment variables allow us to keep sensitive information such as API keys separate from the source code.

Example:

```env
GROQ_API_KEY=your_api_key_here
```

```python
my_api_key = os.getenv("GROQ_API_KEY")
```

## Q11. What does `load_dotenv()` do?

`load_dotenv()` loads environment variables from a `.env` file so they can be accessed in Python using functions such as `os.getenv()`.

## Q12. How do we extract the generated answer from the LLM response?

In the Groq Chat Completions API used in this project:

```python
answer = response.choices[0].message.content
```

- `choices` → Contains the generated response choices.
- `[0]` → Selects the first generated response.
- `message` → Contains the generated message.
- `content` → Contains the actual text answer.
